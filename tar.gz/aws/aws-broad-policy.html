Standard Policy Structure

All AWS resource policies follow a similar JSON structure:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "whatever description",
            "Effect": "Allow/Deny",
            "Principal": {}, // who
            "Action": [],    // do what
            "Resource": "",
            "Condition": {} // match equal? not equal?
        }
    ]
}

Common Policy Elements

Effect

Allow or Deny

Action

Service-specific actions

  • s3:GetObject
  • sqs:SendMessage

Resource

ARN of the resource

Condition

something like if-else string matches or not operator?

Principal

  1. Don't use Principal in IAM Policies (since they're already attached to specific IAM identities) You can't use the Principal element in identity-based policies - those permissions are already attached to specific IAM users, groups, or roles, so the principal is implicit.
    You cannot use the Principal element in an identity-based policy. Identity-based policies are permissions policies that you attach to IAM identities (users, groups, or roles). In those cases, the principal is implicitly the identity where the policy is attached.

    https://docs.aws.amazon.com/IAM/latest/UserGuide/reference_policies_elements_principal.html
  1. Resource Policies must specify Principal
  1. CloudFormation Stack Policy: No Principal needed - it's mainly for controlling update behavior, not service-level stuff
